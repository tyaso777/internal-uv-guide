uv の基本操作と概念

uv を使ううえで必要になる主要概念、基本コマンド、依存関係の記法を整理します。

ページ概要

このページの目的は、uv を使ううえで必要になる基本概念、主要コマンド、依存関係の書き方を整理することです。実際の作業手順は、新規 uv プロジェクトの始め方 や clone・コピーした uv プロジェクトの始め方 で扱います。

主要概念

概念説明
pyproject.tomlプロジェクトのメタデータ、依存関係、ツール設定を置く標準ファイル。
uv.lock解決済み依存関係を固定する lock file。
.python-versionプロジェクトで使う Python バージョンを指定するファイル。
.venvプロジェクトローカルの仮想環境。標準ではプロジェクトフォルダ直下に作成される。
UV_PROJECT_ENVIRONMENTプロジェクト仮想環境の作成先を変更する環境変数。共有フォルダ上に .venv を置きたくない場合に使う。
uv run必要に応じて環境を同期し、仮想環境内でコマンドを実行する。

新規プロジェクト

$projectName には作成したいプロジェクトフォルダ名を指定します。C:\path\to\your\project-parent-folder は例であり、プロジェクトを配置する親フォルダのパスに置き換えます。

仮想環境名をプロジェクト名だけで決めると、別の場所にある同名プロジェクトと衝突する可能性があります。そのため、プロジェクトの保存場所から短いハッシュを作り、仮想環境名の末尾に付けます。

powershell
# プロジェクト名を設定する
$projectName="my-app"

# プロジェクトを配置する親フォルダへ移動する
cd C:\path\to\your\project-parent-folder

# 現在のフォルダの下にプロジェクトを作成する
uv init $projectName

# 作成されたプロジェクトフォルダへ移動する
cd $projectName

# 同じプロジェクト名の仮想環境と衝突しないように、
# プロジェクトの保存場所をもとに短いハッシュを作る
$projectRoot=(Resolve-Path .).Path.ToLowerInvariant()
$sha256=[System.Security.Cryptography.SHA256]::Create()
$hashBytes=$sha256.ComputeHash([System.Text.Encoding]::UTF8.GetBytes($projectRoot))
$hash=([System.BitConverter]::ToString($hashBytes) -replace "-", "").Substring(0,8).ToLowerInvariant()

# 仮想環境の作成先を、このプロジェクト用のローカルフォルダに設定する
$env:UV_PROJECT_ENVIRONMENT="$env:USERPROFILE\.uv\venvs\$projectName-$hash"

# 設定された仮想環境の場所を確認する
echo $env:UV_PROJECT_ENVIRONMENT

# プロジェクトで使う Python バージョンを固定する
uv python pin 3.12

# 依存関係を追加して、プロジェクト環境で実行する
uv add requests
uv run python -c "import requests; print(requests.__version__)"

uv のコマンドは、ファイルを新規作成する場合と、既存ファイルを更新する場合があります。初回実行時に作成されるものと、既に存在する場合に更新されるものを分けて確認します。

コマンド起きること作成されるもの更新されるもの
uv init $projectName現在のフォルダの下に $projectName で指定したプロジェクトを作成する。プロジェクトフォルダ、pyproject.toml、README.md、サンプルコードなど。通常は既存ファイルを更新しない。既存フォルダを指定する場合は内容を確認する。
uv python pin 3.12このプロジェクトで使う Python バージョンを指定する。.python-version が無い場合は作成する。既に .python-version がある場合は更新する。
uv add requestsrequests を依存関係に追加し、依存解決を行う。uv.lock が無い場合は作成する。必要に応じて .venv を作成する。pyproject.toml と uv.lock を更新する。既に .venv がある場合は環境も更新する。

依存関係の追加と削除

powershell
uv add pandas
uv add "fastapi[standard]"
uv add --dev pytest ruff
uv remove pandas
uv lock

pyproject.toml のライブラリ記法

uv の依存管理では、uv add で指定した内容が pyproject.toml に記録され、解決結果が uv.lock に固定されます。レビュー時は、lock file だけでなく pyproject.toml にどのような制約が入ったかを確認します。

基本は PEP 508 の dependency specifier です。ライブラリ名、バージョン制約、extras、環境マーカーを 1 つの文字列として書きます。

toml
[project]
name = "my-app"
version = "0.1.0"
requires-python = ">=3.11,<3.13"
dependencies = [
    "pandas>=2.2,<3",
    "requests>=2.32",
    "fastapi[standard]>=0.115",
    "pywin32>=306; sys_platform == \"win32\"",
]
Warning
手で pyproject.toml を編集することもできますが、通常は uv add を使います。uv add は依存定義と lock file の更新を同時に行うため、記法ミスや lock file の更新漏れを減らせます。

バージョン制約の書き方

指定例使いどころ
下限のみpandas>=2.2基本形。新しい互換版を許容する。
範囲指定pandas>=2.2,<3メジャーバージョンをまたぐ破壊的変更を避けたい場合。
完全固定pandas==2.2.3一時的な回避策や再現性が強く必要な場合。通常は lock file 固定を優先する。
除外pandas!=2.2.2特定バージョンに既知不具合がある場合。
互換リリースpandas~=2.2PEP 440 の互換リリース指定。チームで意味を統一してから使う。

アプリケーションでは >=下限,<次のメジャー のように許容範囲を明示し、実際に使う正確なバージョンは uv.lock で固定します。

Poetry の ^ 記法との対応

Poetry では ^2.2 のような caret requirement をよく使います。これは Poetry 独自の便利な記法で、Python 標準の PEP 508 文字列として [project].dependencies にそのまま書くものではありません。uv 管理へ移行する場合は、意味が分かる範囲指定へ展開して書きます。

Poetry 記法展開後の例意味
^2.2>=2.2,<32.x 系を許容し、3.0.0 以上は避ける。
^2.2.3>=2.2.3,<32.2.3 以上の 2.x 系を許容する。
^0.5.1>=0.5.1,<0.60.x 系では互換範囲が狭くなる点に注意する。
^0.0.3>=0.0.3,<0.0.40.0.x では patch 範囲だけを許容する。
^1.0.0>=1.0.0,<21.x 系を許容する。
Warning
~=2.2 は PEP 440 の compatible release 指定ですが、Poetry の ^ と常に同じ意味ではありません。社内標準では、移行時に ^ を機械的に ~= へ置き換えず、>=下限,<上限 の形で明示します。

uv add での指定方法

uv add でも、基本的には pyproject.toml に入る文字列と同じ指定を使います。PowerShell では <、>、;、extras を含む指定が解釈されないよう、依存指定全体を引用符で囲みます。

powershell
uv add "pandas>=2.2,<3"
uv add "requests>=2.32"
uv add "fastapi[standard]>=0.115"
uv add "pywin32>=306; sys_platform == \"win32\""
uv add --dev "pytest>=8" ruff
uv add --optional notebook "jupyterlab>=4"

上記の結果、runtime dependency は [project].dependencies に、開発用依存は [dependency-groups].dev に、optional dependency は [project.optional-dependencies] に入ります。

toml
[dependency-groups]
dev = [
    "pytest>=8",
    "ruff",
]

[project.optional-dependencies]
notebook = [
    "jupyterlab>=4",
]

仮想環境を C ドライブ内に分離する

uv のプロジェクト仮想環境は、標準ではプロジェクトフォルダ直下の .venv に作成されます。プロジェクトを共有フォルダ上で扱う場合、ネットワーク越しの大量ファイルアクセスを避けるため、仮想環境だけをローカルディスクへ配置できます。

UV_PROJECT_ENVIRONMENT を設定すると、uv が作成・利用するプロジェクト仮想環境の場所を変更できます。次の例では、pyenv-win の配置に近い考え方として、ユーザーフォルダ配下の .uv\venvs にこのプロジェクト用の仮想環境を作成します。

仮想環境を集約するフォルダは $env:USERPROFILE\.uv\venvs です。仮想環境名は プロジェクト名-パス由来ハッシュ の形にします。これにより、同じプロジェクト名でも保存場所が異なる場合は別の仮想環境になります。

powershell
# プロジェクト名を設定する
$projectName="my-app"

# 同じプロジェクト名の仮想環境と衝突しないように、
# プロジェクトの保存場所をもとに短いハッシュを作る
$projectRoot=(Resolve-Path .).Path.ToLowerInvariant()
$sha256=[System.Security.Cryptography.SHA256]::Create()
$hashBytes=$sha256.ComputeHash([System.Text.Encoding]::UTF8.GetBytes($projectRoot))
$hash=([System.BitConverter]::ToString($hashBytes) -replace "-", "").Substring(0,8).ToLowerInvariant()

# このプロジェクト用の仮想環境をユーザーフォルダ配下に配置する
$env:UV_PROJECT_ENVIRONMENT="$env:USERPROFILE\.uv\venvs\$projectName-$hash"

# 設定された仮想環境の場所を確認する
echo $env:UV_PROJECT_ENVIRONMENT

# 指定した場所に仮想環境を作成・同期する
uv sync
Warning
UV_PROJECT_ENVIRONMENT に固定の絶対パスを指定して複数プロジェクトで使い回すと、仮想環境の内容が上書きされる可能性があります。社内手順では、プロジェクト名とパス由来ハッシュを含むパスを使います。

UV_PROJECT_ENVIRONMENT はプロジェクトごとに異なる値を指定する必要があります。そのため、Windows のユーザー環境変数として永続化する運用は標準にはしません。プロジェクトごとの作業開始手順またはセットアップスクリプトで、この PowerShell セッションに設定します。

extras・markers・特殊な依存

9.1 extras

extras は、パッケージが提供する追加依存をまとめて入れる指定です。例: fastapi[standard]、requests[socks]。

powershell
uv add "fastapi[standard]>=0.115"

9.2 environment markers

OS や Python バージョンで依存を分ける場合は environment marker を使います。

toml
dependencies = [
    "pywin32>=306; sys_platform == \"win32\"",
    "tomli>=2; python_version < \"3.11\"",
]

9.3 direct URL / Git / path

通常の社内開発では package index からの依存を優先します。Git、URL、path 依存は再現性、認証、監査、ライセンス確認が難しくなるため、必要な場合だけ使います。

powershell
uv add "my-package @ git+https://github.com/example/my-package.git@v1.2.3"
uv add "my-lib @ file:///${PWD}/libs/my-lib"
Warning
Git や path 依存を使う場合は、参照先、タグまたは commit、認証方式を必ず確認します。社内標準では社内パッケージリポジトリへ配布してから通常依存として追加する方がレビューしやすいです。
