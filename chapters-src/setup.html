uv のネットワーク設定の基本

社内 proxy、証明書、社内パッケージリポジトリなど、uv のネットワーク設定の考え方を整理します。

ページ概要

このページの目的は、社内ネットワーク配下で uv を安定して利用するために必要な環境変数と設定の考え方を整理することです。社内 proxy、SSL inspection、社内 CA、社内パッケージリポジトリを利用する環境を前提に、HTTP_PROXY、HTTPS_PROXY、UV_SYSTEM_CERTS などの設定方法を説明します。

実際の作業手順は、新規 uv プロジェクトの始め方 と clone・コピーした uv プロジェクトの始め方 で改めて記載します。このページでは、各設定が何のためのものか、どこに保存されるか、どの範囲に効くかを押さえます。

proxy の確認

Windows 端末で OS 側に設定されている WinHTTP proxy を確認します。このコマンドは設定を変更せず、現在の端末設定を表示するだけです。

powershell
netsh winhttp show proxy

設定の有効範囲

uv に渡す設定は、どこに保存するかで有効範囲が変わります。トラブル時に原因を追えるよう、手順書やセットアップスクリプトでは「このセッションだけ」なのか「永続設定」なのかを明示します。

設定方法には、PowerShell でコマンドとして実行するものと、設定ファイルに記載するものがあります。下表では、設定方法欄の先頭にどちらの操作かを示します。

設定方法 有効範囲 保存場所 確認方法
PowerShell で一時設定
$env:NAME="VALUE"		 現在の PowerShell セッションのみ 現在のプロセス環境。PowerShell を閉じると消える。 echo $env:NAME
PowerShell で永続設定
[Environment]::SetEnvironmentVariable("NAME", "VALUE", "User")		 現在の Windows ユーザーで永続 Windows のユーザー環境変数。新しく開いた PowerShell から反映される。 [Environment]::GetEnvironmentVariable("NAME", "User")
プロジェクト設定ファイルに記載
pyproject.toml の [tool.uv]		 そのプロジェクト配下 プロジェクトの pyproject.toml ファイルをレビューする
uv 設定ファイルに記載
uv.toml		 配置場所に応じてプロジェクト / ユーザー / システム プロジェクトの uv.toml、または Windows では %APPDATA%\uv\uv.toml ファイルをレビューする
Note
同じ設定が複数箇所にある場合、コマンドライン引数、環境変数、永続設定ファイルの順に優先されます。検証時は一時的な $env: 設定が残っていないか確認します。

proxy 環境変数

この PowerShell セッションだけで試す場合は、$env:HTTP_PROXY と $env:HTTPS_PROXY を設定します。値は PowerShell を閉じると消えます。

powershell
# proxy をこの PowerShell セッションに設定する
$env:HTTP_PROXY="http://proxy.example.co.jp:8080"
$env:HTTPS_PROXY="http://proxy.example.co.jp:8080"

# 設定された proxy を確認する
echo $env:HTTP_PROXY
echo $env:HTTPS_PROXY

次回以降の PowerShell でも使う場合は、Windows のユーザー環境変数として保存します。保存後、既に開いている PowerShell には自動反映されないため、新しい PowerShell を開いて確認します。

powershell
# proxy を Windows のユーザー環境変数として保存する
[Environment]::SetEnvironmentVariable("HTTP_PROXY", "http://proxy.example.co.jp:8080", "User")
[Environment]::SetEnvironmentVariable("HTTPS_PROXY", "http://proxy.example.co.jp:8080", "User")

# 新しい PowerShell を開き、保存された proxy が反映されていることを確認する
echo $env:HTTP_PROXY
echo $env:HTTPS_PROXY

社内 CA / SSL inspection

社内 proxy が TLS 通信を検査している場合、OS の証明書ストアに入っている社内ルート CA を uv に利用させます。まずは現在の PowerShell セッションだけで有効化して、通信できるか確認します。

powershell
# OS の証明書ストアをこの PowerShell セッションで利用する
$env:UV_SYSTEM_CERTS="1"

# 設定された値を確認する
echo $env:UV_SYSTEM_CERTS

継続して必要な端末では、Windows のユーザー環境変数として永続化します。保存場所は Windows のユーザー環境変数です。

powershell
# OS の証明書ストアを使う設定を Windows のユーザー環境変数として保存する
[Environment]::SetEnvironmentVariable("UV_SYSTEM_CERTS", "1", "User")

# 新しい PowerShell を開き、保存された値が反映されていることを確認する
echo $env:UV_SYSTEM_CERTS

社内パッケージリポジトリ

社内パッケージリポジトリは、社内で作成した Python パッケージや、社内でミラーした外部パッケージを配布するためのパッケージ置き場です。PyPI ではなく社内のパッケージリポジトリから取得する必要があるプロジェクトで使います。

すべてのプロジェクトで設定するものではありません。公開 PyPI から依存パッケージを取得できるプロジェクトや、社内 private package を使わないプロジェクトでは、この設定は不要です。

社内パッケージリポジトリをプロジェクトごとに固定する場合は、プロジェクトの pyproject.toml に保存します。この設定はリポジトリに含まれるため、レビューと履歴管理の対象になります。

toml
[[tool.uv.index]]
name = "internal"
url = "https://packages.example.co.jp/simple"
default = true

端末全体の既定値として保存する場合は、Windows では %APPDATA%\uv\uv.toml を使います。ユーザー設定に入れると、複数プロジェクトへ影響するため、社内標準として配布する設定だけにします。

toml
[[index]]
name = "internal"
url = "https://packages.example.co.jp/simple"
default = true

新しすぎるリリースを避ける

依存パッケージの公開直後は、破損した wheel、差し戻し、悪意あるパッケージ混入、社内 mirror への反映遅延が起きる可能性があります。

powershell
# コマンド実行時に、直近 7 日以内に公開された依存パッケージを避ける
uv lock --exclude-newer "7 days"
uv sync --exclude-newer "7 days"

# この PowerShell セッションの既定値として設定する
$env:UV_EXCLUDE_NEWER="7 days"

# 設定された値で lock / sync を実行する
uv lock
uv sync

X=7 とする場合は、--exclude-newer "7 days" または UV_EXCLUDE_NEWER="7 days" を指定します。$env:UV_EXCLUDE_NEWER は現在の PowerShell セッションだけに効きます。永続化する場合はユーザー環境変数として保存できますが、lock file 更新の意図が見えにくくなるため、社内標準ではコマンドライン引数で明示します。
